Git Basics on Linux

Create the Repository

  1. At GitHub website, create a repository, adding README file.

  2. Copy/clone this repository to local machine clicking the green "Code" button. Select ths SSH option and copy the URL below it.

  3. Use local machine CL(command line) to create a new directory for the project. Home folder is represented by ~. If not sure if you're in the home folder, type cd ~ . 

              
            mkdir directoryName
            cd directoryName/

  4. Now clone your repository from GitHub onto local machine with git clone followed by the URL copied last in the step. Look similar to git clone git@github.com:USER-NAME/REPOSITORY-NAME.git. 

                  
                git clone git@github.com:USER-NAME/REPOSITORY-NAME.git

  5. That's it! Connected the Github repository to local machine. 

                  
                git remote -v  // Display the repo URL

Use Git Workflow

  1. Create a new file in the project folder. 

              
            touch newfile //edit the newfile to desired filename

  2. Type git status if shown in red, file is not staged.

  3. Type git add [filename] adds to the staging area in Git.

    Think of the staging area as a "waiting room" for the changes until commit them.

    Type git status. The output shown in green, means it is now in the staging area.

  4. Type git commit -m "Add [filename]" and them type git status to see the status.

  5. Type git log to see an entry for the recent commit. Author, date, time. If terminal become stuck, press "q" to escape.

Modify a file or two

  1. Open the file in the editor of choice. Here we use VS Code by using the command code . inside the repository.

  2. Edit the file and save with "Ctrl+S".

  3. Add the file to the staging area with git add [filename]

    Use git add . to add all files in the current directory and subsequent directories to the staging area.

  4. Finally commit all the files and add a descriptive commit message. 

                
              git commit -m "Here goes the desired message in quotation marks"
    Then type git status to see the output.

  5. Take a last look at the commit history typing git log.

Push the work to GitHub

Finally upload the work to the GitHub repository.

  1. Type git push or if have more branches, git push origin main (change arguments if necessary).

  2. Type git status to see the status(Optional).

Avoid changes directly via GitHub, it is advised to make any changes via local files then commit and push using Git commands.

Cheatsheet

Commands related to a remote repository:

    git clone git@github.com:USER-NAME/REPOSITORY-NAME.git

    git push or git push origin main (Both accomplish the same goal in this context)

Commands related to the workflow:

  
    git add .

    git commit -m "A message describing what you have done to make this snapshot different"

Commands related to checking status or log history

  
    git status

    git log

The basic Git syntax is program | action | destination.

For example,


  git add . is read as git | add | ., where the period represents everything in the current directory;
  git commit -m "message" is read as git | commit -m | "message"; and
  git status is read as git | status | (no destination).

Commit Messages

Read this article: How to Write a Git Commit Message, covers the main bases.

  1. Separate subject from body with a blank line - e.g. Fix typo in introduction to user guide
  2. Limit the subject line to 50 characters. Hard limit 72, it will truncate with ellipsis if exceed
  3. Capitalize the subject line
  4. Do not end the subject line with a period
  5. Use the imperative mood in the subject line - e.g. Refactor subsystem X for readability, Remove deprecated methods

    A properly formed Git commit subject line should always be able to complete the following sentence:

    If applied, this commit will your subject line here

    • If applied, this commit will refactor subsystem X for readability
    • If applied, this commit will update getting started documentation
    • If applied, this commit will remove deprecated methods
    • If applied, this commit will release version 1.0.0
    • If applied, this commit will merge pull request #123 from user/branch
  6. Wrap the body at 72 characters
  7. Use the body to explain what and why vs. how

Tips and things to Remember

  • Use an active voice: “Fix card generator”
  • Avoid using vague commit messages such as “saved” or “updated”
  • Commit early and often!

Git Best Practices

  • Atomic commits - changes related only one feature or task of the program. Easy to revert without losing other changes and enables to write better commit messages.

  • Meaningful commit messages - help understand better the changes.

Changing the Git Commit Message Editor

If you are using Visual Studio Code (and you should be if you’re following this curriculum) and you don’t want to get stuck writing a commit message in Vim because you accidentally used git commit without the message flag (-m), this command will make Visual Studio Code open a new tab with the ability to write your commit message and an optional description below it: git config --global core.editor "code --wait".

There will be no confirmation or any output on the terminal after entering this command. To make a commit with Visual Studio Code as the text editor, make sure to use the git commit command without the -m flag. Just type git commit and no message after that. Once you do this, a new tab will open. Now you can write your message, and provide more information if you want, right below it. After typing your commit message, save it and exit the tab.

With that out of the way, now you can choose to use either git commit -m [your message here] or git commit and enter your message with Visual Studio Code!
